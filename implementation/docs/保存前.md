## 2. 設計メモ

### 2-1. 文書境界
- この文書は `要求差分` と `設計メモ` を同居させるが、混線を防ぐために章を分離する。
- 要求差分には `何を達成するか / 何のためか` だけを書く。
- 実装方式、構造、用語定義、代表シナリオ、採用理由、今はやらない話は設計メモか後段のメモへ置く。

### 2-2. 入力分離の具体化
- BOT.01-04 の目的は、bot が business intent を先に分類することではなく、入力の出自を混ぜないことである。
- 各メッセージで渡す情報は、少なくとも次の 3 種に分けて扱う。
  - 利用者本文とその直近メッセージ事実
  - system 制約と返信先に関わる事実
  - 会話場所、thread lineage、source message 対応のような Discord 上の事実
- 内部知見への入口は、各メッセージごとに候補一覧を押し込むのではなく、`AGENTS.md` と Skill 側で固定 read interface として最初に与える。
- system は「何を読むべきか」を決めない。system が決めるのは scope と権限上の参照可能境界だけであり、実際に何を読むかは LLM が判断する。
- Discord の ID や場所情報は、名前だけ渡しても意味が足りないため、`AGENTS.md` と Skill が「その ID が何を指し、どう扱うか」を補う。

### 2-3. 解釈と境界の責務分担
- system 側に置くもの
  - private/public 境界
  - scope filter
  - reply target 固定
  - thread lineage
  - 重複防止
  - 保存整合性
  - timeout / retry / observability
- Skill / LLM 側に置くもの
  - Discord 文脈の意味解釈
  - URL を読むべきかの判断
  - 既存 source で足りるかの判断
  - 追加取得手段の選択
  - 要約
  - 補強
  - 回答文

### 2-4. Harness と script / tool の境界
- この文書では Harness の詳細定義を展開しない。Harness の原則は `/agents-harness-implementation` を前提にし、この repo で常設コンテキストとして扱うものは `AGENTS.md` と Skill だけにする。
- Skill は Harness の一部である。
- script、CLI wrapper、DB helper は Harness ではない。Skill が必要時に使う通常ツールである。

### 2-5. script / tool 採用の判断原則
- script や補助ツールを置くのは、次の 3 条件を満たすときだけにする。
  1. `LLM にやらせる価値のない単純な作業か`
  2. `LLM による揺らぎを産まずに必ず定量的・決定的に処理したい内容か`
  3. `LLM にやらせたほうが柔軟で意義深いという本末転倒にならないか`
- これに従い、`URL 安全判定`, `scope filter`, `reply target 固定`, `dedupe`, `event logging` は system/tool 側に残す。
- `検索戦略`, `取得手段選択`, `要約`, `補強`, `回答文` は Skill / LLM 側に残す。

### 2-6. URL 境界の用語
- 設計上は URL を少なくとも `外部取得可能な公開 URL` と `取得禁止 URL` に分ける。
- `取得禁止 URL` に含める対象は、LLM 解釈ではなく system が決定的に判定できるものに限る。
  - `file:` `data:` `javascript:` のような実行可能またはローカル資源参照スキーム
  - `localhost`, `127.0.0.1`, `::1`
  - RFC1918 private IP、link-local、`.local` などの私設到達先
  - リダイレクト後に上記へ到達する URL
- この判定を system に置く理由は、prompt injection 判定のような意味解釈ではなく、SSRF とローカル資源漏洩防止を文字列・DNS・IP 範囲で定量的に扱えるからである。
- この分類は「何の intent か」を決めるためではなく、「何を外部取得させてよいか」を境界として示すために使う。

### 2-7. 重複処理防止の扱い
- RUN.01-04 の具体設計では、live 受信と再開処理が同じ message を運んでも、処理開始は 1 回だけになる仕組みを持つ。
- その仕組みとして claim を使うことはあり得るが、claim 自体は要求ではなく設計である。

### 2-8. 一時緩和の境界
- AUTH.03-02 の要求は「適用範囲と有効期間を限定すること」である。
- `何を緩和できるか`, `v1 に含めない運用`, `コード変更を自動確定しない` のような話は要求ではなく設計上の境界条件として扱う。

## 3. 検証観点

### 3-1. 要求差分のレビュー観点
- 各要求文を単独で読んだとき、そこから仕様論点を 3 つ以上導けること。
- 要求文に field 名、tool 名、script 名、API 名、Playwright 名、編集メモが残っていないこと。
- `BOT.01-04/05/06/07`, `RUN.01-04`, `ING.01-02`, `AUTH.03-02` が、要求・設計・メモのどこに属するか一意に説明できること。

### 3-2. 代表シナリオ
- 通常の plain chat
- URL 投稿
- URL を含む投稿から作られた知見スレッド内 follow-up
- 管理者による一時緩和
- `codex turn timed out` を含む失敗

### 3-3. Harness 観点
- Harness に関する記述が `AGENTS.md` と `Skills` のみに限定されていること。
- script が Harness と読める文が残っていないこと。
- Discord の ID 群だけを渡せばよい、という記述がなく、`AGENTS.md` と Skill が意味づけを補うことが書かれていること。
- 内部知見への入口が「各メッセージごとの候補一覧」ではなく、「最初に与える固定 read interface」として書かれていること。

### 3-4. 境界観点
- `外部取得可能な公開 URL` と `取得禁止 URL` の区別が「解釈分類」ではなく「取得境界」であると読めること。
- シナリオ列挙が分類体系や将来拡張の拘束になっていないこと。
- timeout、retry、重複防止の記述が「何のためか」に結びついていること。

## 4. 発言/確認メモ

- 「IDをどう活用するかについてSkills/docsを渡さないと、多分難しい」

- 「Skillsでスクリプトは置いて置けるけどDBは使えねぇよ」
- 「Skillsはハーネスだろうが」
- 「必須を外すってのが要求として書かれているの意味わからない」
- 「V1に含めないをv1の要求に書くなメモだろ」

### 確認メモ
- `AGENTS.md` の bot runtime 向け更新はこの差分文書の対象外。
- 取得手段を固定しない方針は要求差分で保持するが、具体候補の比較や既定値は設計メモ側でのみ扱う。
- 将来、要求差分が増える場合も、要求節には「what / why」だけを書き、技術選定や構造は設計メモへ退避する。
