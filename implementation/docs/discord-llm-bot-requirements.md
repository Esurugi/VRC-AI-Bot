# Discord LLM Bot 要求整理

## 要求テーブル

### 【全体利用】

| ID | 要求文 |
|---|---|
| BOT.01 | Discord サーバー参加者が、サーバー内の投稿を通じて LLM bot を利用できるようにする。 |
| 理由 [引用] | 「既存のコミュニティ用DiscordにLLMが導入されたbotを入れたい」 |
| BOT.01.01 | サーバー内の投稿内容を取得する。 |
| BOT.01.02 | 投稿内容から URL または質問を抽出する。 |
| BOT.01.03 | 抽出結果に応じて知見整理または応答を実行する。 |
| BOT.01.04 | bot runtime は、LLM が DB 読み出しや追加 Discord facts 取得の手順を実装読解なしで使える local skill / script contract を提供し、Codex session は返信先ではなく session identity で管理する。 |
| BOT.01.05 | System は意味解釈や検索語決定を持ちすぎず、LLM が save intent、retrieval strategy、query wording を主導できるようにする。 |
| BOT.01.06 | System は Codex session を place 単位の文字列キーではなく、workload/binding/authority/version を含む session identity で管理し、runtime contract が変わった旧 session を resume しない。 |

### 【停止中からの再開】

| ID | 要求文 |
|---|---|
| RUN.01 | bot 起動時に、停止中に投稿された未処理の対象メッセージを取得して受付順に処理する。 |
| 理由 [引用] | 「僕が自分のPCから起動した時にそれを取得して順番に処理していってほしい」 |
| RUN.01.01 | 停止中に投稿された対象メッセージを過去ログから抽出する。 |
| RUN.01.02 | 抽出した対象メッセージを受付順に並べる。 |
| RUN.01.03 | 並べた対象メッセージを順次処理する。 |
| RUN.01.04 | transient runtime failure の対象メッセージは DB-backed retry scheduler に載せ、`message_processing` を `pending_retry` として保持したまま再試行できるようにする。 |
| RUN.01.05 | `pending_retry` または `processing` の重複処理では cursor を進めず、成功または終端失敗で `completed` になったときだけ前進する。 |

### 【URL知見化】

| ID | 要求文 |
|---|---|
| ING.01 | サーバー内の投稿に含まれる公開 Web ページの URL から内容を取得する。 |
| 理由 [要約] | 「良かったサイトのリンクとかを貼ってくれる」「公開 Web ページ全般」 |
| ING.01.01 | URL 先のページ内容を取得する。 |
| ING.01.02 | 取得したページ内容から共有本文を生成する。 |
| ING.01.02a | 非日本語の公開記事は、日本語で読める共有本文を生成する。 |
| ING.01.03 | 要約に検索用タグを付与する。 |
| ING.01.04 | URL、要約、タグを後から参照できる形で保持する。 |
| ING.01.04a | 利用者が自然文で明示的に保存を依頼した場合、貼り付け URL がなくても外部公開情報を取得して保持できるようにする。 |
| ING.01.04b | 自然文による明示保存は、保存元の会話場所に関係なく同一 guild の `server_public` 知見として保持する。 |

### 【知見活用】

| ID | 要求文 |
|---|---|
| THR.01 | bot は knowledge thread に限らず、現在の会話場所から見える蓄積知見を参照して参加者の質問に応答する。 |
| 理由 [要約] | 「Threadでなくても共有知見は普通の雑談や質問とかでもデータを引っ張ってくれて良い」 |
| THR.01.01 | URL 知見共有では、元投稿に対応するスレッドを作成する。 |
| THR.01.02 | 会話場所ごとの質問内容を取得する。 |
| THR.01.03 | 現在の会話場所から見える蓄積済み知見と、必要に応じて元の URL 内容を参照して回答を生成する。 |
| THR.01.04 | 既存知見で不足する場合に限り、外部公開情報を参照して回答を補強する。 |
| THR.01.05 | 生成した回答を会話場所に応じて返す。 |
| THR.01.05a | スレッド内の人間による follow-up は無言で終了させず、same thread に可視応答を返す。 |
| THR.01.05b | 通常 chat や admin_control の root 会話でも、見えている知見が関連する場合は same place で知見を使った応答を返す。 |

### 【雑談利用】

| ID | 要求文 |
|---|---|
| CHAT.01 | 雑談用の特定チャンネルでは、参加者との通常会話に応答する。 |
| 理由 [引用] | 「ただのチャット相手としての運用は、それ用途の特定チャンネルを作る」 |
| CHAT.01.01 | 雑談用チャンネル内の発話内容を取得する。 |
| CHAT.01.02 | 発話内容に応じた応答を生成する。 |
| CHAT.01.03 | 生成した応答をチャンネル内に返す。 |
| CHAT.01.04 | 雑談用チャンネルでは、URL を含む投稿もまず通常会話として扱い、自動で知見共有 thread の作成や知見保存を行わない。 |
| CHAT.01.05 | 雑談用チャンネルでも、見えている蓄積知見は通常会話の補助情報として参照してよい。 |

### 【機密区分】

| ID | 要求文 |
|---|---|
| SEC.01 | bot は、投稿場所と投稿内容の機密区分に応じて、参照可能な情報範囲と出力可能な内容を制御する。 |
| 理由 [引用] | 「非公開チャンネルや管理者向けチャンネルで取得した内容は、公開チャンネルに漏れないようにする必要がある」 |
| SEC.01.01 | 投稿場所の公開範囲から機密区分を判定する。 |
| SEC.01.02 | 投稿内容に機密区分を付与する。 |
| SEC.01.03 | 機密区分ごとに参照可能な保持データを制限する。 |
| SEC.01.04 | 公開先に返す内容から上位機密区分の情報を除外する。 |
| SEC.01.05 | 非公開由来の内容を明示的な許可なしに公開範囲へ出力しない。 |
| SEC.01.04a | system は最終出力直前に `sources_used` と source visibility を検査し、scope 外 source や blocked/private URL を根拠にした応答をそのまま送信しない。 |
| SEC.01.04b | source 境界違反があった場合、system は 1 回だけ安全再生成を要求し、なお unsafe なら same place / same thread に短い refusal を返す。 |
| SEC.01.05a | `conversation_only` や admin_control 由来 source を `channel_family` / `server_public` へそのまま出さず、同じ事実が公開可視 source にあるか同 turn の公開再確認がある場合に限って許可する。 |
| SEC.01.05b | same-turn public reconfirmation の根拠は、repo-local skill `public-source-fetch` の構造化出力だけを authoritative evidence として扱う。 |
| SEC.01.05c | knowledge thread の non-empty follow-up は無言終了させず、system は 1 回だけ control-plane retry を要求し、それでも不可なら generic same-thread failure を返す。 |

### 【権限制御】

| ID | 要求文 |
|---|---|
| AUTH.01 | bot は、あなたまたは管理者以外の利用者から与えられた指示によって、運用方針、機密区分、参照範囲、または内部動作方針を変更しない。 |
| 理由 [引用] | 「僕以外の管理者以外の人間からのプロンプトインジェクション対策を実施したい」 |
| AUTH.01.01 | 管理者を Discord の Administrator 権限保持者として識別する。 |
| AUTH.01.02 | 投稿内容に含まれる指示のうち、通常応答の範囲を超える変更要求を識別する。 |
| AUTH.01.03 | あなたまたは管理者以外の利用者からの変更要求を運用指示として採用しない。 |
| AUTH.01.04 | 通常利用者の投稿内容を、知見取得や会話応答の対象としてのみ扱う。 |
| AUTH.01.05 | 通常利用者の投稿内容によって上位機密区分の情報参照や出力範囲拡大を行わない。 |
| AUTH.01.06 | 通常応答の範囲内に収まる無害な表現変更の依頼は受け付ける。 |

| ID | 要求文 |
|---|---|
| AUTH.02 | bot は、一般利用者からの禁止指示または危険な指示の繰り返しを利用者単位で記録し、しきい値に応じて利用制限を適用する。 |
| 理由 [要約] | 「内部でカウンターを用意して、ユーザーIDなど識別できる情報と紐づけておくといい」「あまりにも多かったらタイムアウト」「またやらかしたらキック」 |
| AUTH.02.01 | 禁止指示または危険な指示の発生回数を利用者識別子に紐づけて記録する。 |
| AUTH.02.02 | 記録回数が所定のしきい値を超えた利用者に一時的な利用制限を適用する。 |
| AUTH.02.03 | 一時的な利用制限の後も再発した利用者に、より強い利用制限を適用する。 |

| ID | 要求文 |
|---|---|
| AUTH.03 | 管理者は、必要なときに管理者限定の Discord bot command を使って bot の制約を一時的に緩和し、検証を実行できるようにする。 |
| 理由 [引用] | 「管理者は一時的に制約を解除して試すことができてほしい」 |
| AUTH.03.01 | 管理者限定の Discord bot command による制約緩和要求を識別する。 |
| AUTH.03.02 | 制約緩和 command は Discord の Administrator 権限保持者だけが実行できるようにする。 |
| AUTH.03.03 | 制約緩和の適用範囲を、configured `admin_control` root channel から command で開いた専用 thread 単位に限定して扱う。 |
| AUTH.03.04 | 制約緩和が有効でない通常運用の Codex は read-only sandbox で動作する。 |
| AUTH.03.05 | 制約緩和中の専用 thread では、override を開始した同一管理者の会話全体を workspace-write sandbox で動作させる。 |
| AUTH.03.05a | 制約緩和中の専用 thread では、override を開始した同一管理者に渡す Harness capability を `allow_external_fetch`, `allow_knowledge_write`, `allow_moderation` の全てで true とする。Discord thread 作成は system の reply-routing で扱い、LLM capability には含めない。 |
| AUTH.03.06 | 管理者が専用 thread 内で終了 command を明示実行できるようにし、終了時は thread と workspace-write Codex の両方を閉じて通常の read-only 制約状態へ戻す。 |
| AUTH.03.07 | 制約緩和用 thread と対応する workspace-write Codex session は、session identity の version が一致する間だけ、bot 停止、再起動、container 再作成後も dedicated override thread 単位で再利用できるようにする。 |

### 【Harness契約】

| ID | 要求文 |
|---|---|
| BOT.01.06a | bot は Harness turn を `intent -> answer -> optional retry` の 2 段階基本フローで扱い、System は intent 解釈を追加しない。 |
| BOT.01.06b | `available_context` は facts-only とし、retry 制御情報は `task.retry_context` に分離する。 |
| BOT.01.06c | 外部取得と knowledge 保存の capability は Harness request と System factual gate の両方を満たした `answer` turn にだけ付与する。 |

### 【コンテナ運用】

| ID | 要求文 |
|---|---|
| OPS.01 | bot は VRC-AI-Bot の実行環境を Docker コンテナ内で動作させる。 |
| 理由 [要約] | 「Docker上でCodexにも動いてもらいたい」「Codexのローカルのスキルが覗けないように、ポートが見えないようにしたい」 |
| OPS.01.01 | bot と Codex CLI/App Server は同一コンテナ内で動作する。 |
| OPS.01.02 | Codex は当該リポジトリ全体を作業対象として参照できる。 |
| OPS.01.03 | ホスト側の `~/.codex`、`~/.claude`、個人用スキル群、他リポジトリはコンテナ内に公開しない。 |

| ID | 要求文 |
|---|---|
| OPS.02 | Codex の実行境界は、リポジトリ作業領域と認証・履歴領域を分離して扱う。 |
| 理由 [要約] | 「Codexのローカルのスキルが覗けないように」「チャットボットに話しかけたときの履歴が全部こちらの方に上がってきてしまっている」 |
| OPS.02.01 | Codex の作業ディレクトリと Codex home は別パスで扱う。 |
| OPS.02.02 | Codex home には認証情報と会話履歴を保存できるが、保存先はコンテナ専用領域に限定する。 |
| OPS.02.03 | コンテナ再作成時に Codex home を破棄できるようにする。 |

### 【秘密情報保護】

| ID | 要求文 |
|---|---|
| SEC.02 | bot は秘密情報をリポジトリ内ファイルとして保持せず、Docker 外部から環境変数として受け取る。 |
| 理由 [引用] | 「.envをDockerの外側に置き、環境変数としてのみ渡して、コーデックスのボットがリポジトリを見たときにトークん等を見られないよう、制御したいです。」 |
| SEC.02.01 | `.env` はリポジトリ外に置き、コンテナ内へは環境変数としてのみ注入する。 |
| SEC.02.02 | Codex がリポジトリを読んでも token 等の秘密値をファイル経由で取得できない状態を保つ。 |

### 【Codex認証・履歴隔離】

| ID | 要求文 |
|---|---|
| SEC.03 | Codex の OAuth 認証情報と会話履歴はホスト PC の通常ファイル領域へ残さない。 |
| 理由 [要約] | 「CLIのローカルのChatGPTのOAuth認証を使ってるので、その認証関連もセキュアに出来るならその方が良い」「履歴が全部こちらの方に上がってきてしまっている」 |
| SEC.03.01 | ChatGPT OAuth 認証成果物はコンテナ専用の Codex home にのみ保存する。 |
| SEC.03.02 | Codex の会話履歴、session index、state DB はコンテナ専用の Codex home にのみ保存する。 |
| SEC.03.03 | bot 運用停止やコンテナ再作成時に、Codex home をまとめて破棄または初期化できるようにする。 |

### 【知見保持】

| ID | 要求文 |
|---|---|

### 【失敗時】

| ID | 要求文 |
|---|---|
| ERR.01 | URL 内容の取得または応答処理に失敗したとき、失敗理由と再試行予定を利用者に通知する。 |
| 理由 [引用] | 「もちろん、失敗理由と再試行予定を出す形」 |

### 【ペルソナ】

| ID | 要求文 |
|---|---|
| PER.01 | bot は、参加者に親しみやすい秘書的存在として、一貫した話し方と振る舞いで応答する。 |
| 理由 [要約] | 「親しみやすいようにペルソナを設定した」「AIアシスタントというか秘書だな秘書的存在だと思った方がいい」 |

## 発言メモ

- 「自分がPC起動しているときだけローカルで動かし、Codex App Server経由で動作する想定」

- 「全てのチャンネルのURLでいい」
- 「LLMに与える情報は無駄じゃないものは多いほうが良い」
- 「投稿場所と投稿内容の機密区分で良いと思う」
- 「僕以外の管理者以外の人間からのプロンプトインジェクション対策を実施したい」

- 「チャットボットに話しかけたときの履歴が全部こちらの方に上がってきてしまっているので、それをうまく対処する方法がないか探している」
- 「Windows の `shell:startup` に `start-vrc-ai-bot.bat` のショートカットを置く案はいったん保留」

## 確認メモ

- Discord の権限仕様、チャンネル可視性、スレッド可視性、履歴取得範囲に照らして、`同等以下の公開範囲でのみ再利用する` 制御がどこまで実装しやすいかは確認が必要。
