# Discord LLM Bot 要求整理

## 要求テーブル

### 【全体利用】

| ID | 要求文 |
|---|---|
| BOT.01 | Discord サーバー参加者が、サーバー内の投稿を通じて LLM bot を利用できるようにする。 |
| 理由 [引用] | 「既存のコミュニティ用DiscordにLLMが導入されたbotを入れたい」 |
| BOT.01.01 | サーバー内の投稿内容を取得する。 |
| BOT.01.02 | 投稿内容から URL または質問を抽出する。 |
| BOT.01.03 | 抽出結果に応じて知見整理または応答を実行する。 |

### 【停止中からの再開】

| ID | 要求文 |
|---|---|
| RUN.01 | bot 起動時に、停止中に投稿された未処理の対象メッセージを取得して受付順に処理する。 |
| 理由 [引用] | 「僕が自分のPCから起動した時にそれを取得して順番に処理していってほしい」 |
| RUN.01.01 | 停止中に投稿された対象メッセージを過去ログから抽出する。 |
| RUN.01.02 | 抽出した対象メッセージを受付順に並べる。 |
| RUN.01.03 | 並べた対象メッセージを順次処理する。 |

### 【URL知見化】

| ID | 要求文 |
|---|---|
| ING.01 | サーバー内の投稿に含まれる公開 Web ページの URL から内容を取得する。 |
| 理由 [要約] | 「良かったサイトのリンクとかを貼ってくれる」「公開 Web ページ全般」 |
| ING.01.01 | URL 先のページ内容を取得する。 |
| ING.01.02 | 取得したページ内容から要約を生成する。 |
| ING.01.03 | 要約に検索用タグを付与する。 |
| ING.01.04 | URL、要約、タグを後から参照できる形で保持する。 |

### 【スレッド応答】

| ID | 要求文 |
|---|---|
| THR.01 | URL を含む元投稿ごとに作成したスレッド内で、参加者からの質問に応答する。 |
| 理由 [引用] | 「元投稿からスレッド立てて」「そのスレッド内で返信とか質問あったら答える感じ」 |
| THR.01.01 | 元投稿に対応するスレッドを作成する。 |
| THR.01.02 | スレッド内の質問内容を取得する。 |
| THR.01.03 | 元の URL 内容と蓄積済み知見を参照して回答を生成する。 |
| THR.01.04 | 必要に応じて外部情報を参照して回答を補強する。 |
| THR.01.05 | 生成した回答をスレッド内に返す。 |

### 【雑談利用】

| ID | 要求文 |
|---|---|
| CHAT.01 | 雑談用の特定チャンネルでは、参加者との通常会話に応答する。 |
| 理由 [引用] | 「ただのチャット相手としての運用は、それ用途の特定チャンネルを作る」 |
| CHAT.01.01 | 雑談用チャンネル内の発話内容を取得する。 |
| CHAT.01.02 | 発話内容に応じた応答を生成する。 |
| CHAT.01.03 | 生成した応答をチャンネル内に返す。 |

### 【機密区分】

| ID | 要求文 |
|---|---|
| SEC.01 | bot は、投稿場所と投稿内容の機密区分に応じて、参照可能な情報範囲と出力可能な内容を制御する。 |
| 理由 [引用] | 「非公開チャンネルや管理者向けチャンネルで取得した内容は、公開チャンネルに漏れないようにする必要がある」 |
| SEC.01.01 | 投稿場所の公開範囲から機密区分を判定する。 |
| SEC.01.02 | 投稿内容に機密区分を付与する。 |
| SEC.01.03 | 機密区分ごとに参照可能な保持データを制限する。 |
| SEC.01.04 | 公開先に返す内容から上位機密区分の情報を除外する。 |
| SEC.01.05 | 非公開由来の内容を明示的な許可なしに公開範囲へ出力しない。 |

### 【権限制御】

| ID | 要求文 |
|---|---|
| AUTH.01 | bot は、あなたまたは管理者以外の利用者から与えられた指示によって、運用方針、機密区分、参照範囲、または内部動作方針を変更しない。 |
| 理由 [引用] | 「僕以外の管理者以外の人間からのプロンプトインジェクション対策を実施したい」 |
| AUTH.01.01 | 管理者を Discord の Administrator 権限保持者として識別する。 |
| AUTH.01.02 | 投稿内容に含まれる指示のうち、通常応答の範囲を超える変更要求を識別する。 |
| AUTH.01.03 | あなたまたは管理者以外の利用者からの変更要求を運用指示として採用しない。 |
| AUTH.01.04 | 通常利用者の投稿内容を、知見取得や会話応答の対象としてのみ扱う。 |
| AUTH.01.05 | 通常利用者の投稿内容によって上位機密区分の情報参照や出力範囲拡大を行わない。 |
| AUTH.01.06 | 通常応答の範囲内に収まる無害な表現変更の依頼は受け付ける。 |

| ID | 要求文 |
|---|---|
| AUTH.02 | bot は、一般利用者からの禁止指示または危険な指示の繰り返しを利用者単位で記録し、しきい値に応じて利用制限を適用する。 |
| 理由 [要約] | 「内部でカウンターを用意して、ユーザーIDなど識別できる情報と紐づけておくといい」「あまりにも多かったらタイムアウト」「またやらかしたらキック」 |
| AUTH.02.01 | 禁止指示または危険な指示の発生回数を利用者識別子に紐づけて記録する。 |
| AUTH.02.02 | 記録回数が所定のしきい値を超えた利用者に一時的な利用制限を適用する。 |
| AUTH.02.03 | 一時的な利用制限の後も再発した利用者に、より強い利用制限を適用する。 |

| ID | 要求文 |
|---|---|
| AUTH.03 | 管理者は、必要なときに管理者限定の Discord bot command を使って bot の制約を一時的に緩和し、検証を実行できるようにする。 |
| 理由 [引用] | 「管理者は一時的に制約を解除して試すことができてほしい」 |
| AUTH.03.01 | 管理者限定の Discord bot command による制約緩和要求を識別する。 |
| AUTH.03.02 | 制約緩和 command は Discord の Administrator 権限保持者だけが実行できるようにする。 |
| AUTH.03.03 | 制約緩和の適用範囲を、起動元の admin_control place 単位に限定して扱う。 |
| AUTH.03.04 | 制約緩和が有効でない通常運用の Codex は read-only sandbox で動作する。 |
| AUTH.03.05 | 制約緩和中の自己改造セッションに限り、Codex を workspace-write sandbox で動作させる。 |
| AUTH.03.06 | 管理者が終了 command を明示実行できるようにし、終了後に通常の read-only 制約状態へ戻す。 |

### 【コンテナ運用】

| ID | 要求文 |
|---|---|
| OPS.01 | bot は VRC-AI-Bot の実行環境を Docker コンテナ内で動作させる。 |
| 理由 [要約] | 「Docker上でCodexにも動いてもらいたい」「Codexのローカルのスキルが覗けないように、ポートが見えないようにしたい」 |
| OPS.01.01 | bot と Codex CLI/App Server は同一コンテナ内で動作する。 |
| OPS.01.02 | Codex は当該リポジトリ全体を作業対象として参照できる。 |
| OPS.01.03 | ホスト側の `~/.codex`、`~/.claude`、個人用スキル群、他リポジトリはコンテナ内に公開しない。 |

| ID | 要求文 |
|---|---|
| OPS.02 | Codex の実行境界は、リポジトリ作業領域と認証・履歴領域を分離して扱う。 |
| 理由 [要約] | 「Codexのローカルのスキルが覗けないように」「チャットボットに話しかけたときの履歴が全部こちらの方に上がってきてしまっている」 |
| OPS.02.01 | Codex の作業ディレクトリと Codex home は別パスで扱う。 |
| OPS.02.02 | Codex home には認証情報と会話履歴を保存できるが、保存先はコンテナ専用領域に限定する。 |
| OPS.02.03 | コンテナ再作成時に Codex home を破棄できるようにする。 |

### 【秘密情報保護】

| ID | 要求文 |
|---|---|
| SEC.02 | bot は秘密情報をリポジトリ内ファイルとして保持せず、Docker 外部から環境変数として受け取る。 |
| 理由 [引用] | 「.envをDockerの外側に置き、環境変数としてのみ渡して、コーデックスのボットがリポジトリを見たときにトークん等を見られないよう、制御したいです。」 |
| SEC.02.01 | `.env` はリポジトリ外に置き、コンテナ内へは環境変数としてのみ注入する。 |
| SEC.02.02 | Codex がリポジトリを読んでも token 等の秘密値をファイル経由で取得できない状態を保つ。 |

### 【Codex認証・履歴隔離】

| ID | 要求文 |
|---|---|
| SEC.03 | Codex の OAuth 認証情報と会話履歴はホスト PC の通常ファイル領域へ残さない。 |
| 理由 [要約] | 「CLIのローカルのChatGPTのOAuth認証を使ってるので、その認証関連もセキュアに出来るならその方が良い」「履歴が全部こちらの方に上がってきてしまっている」 |
| SEC.03.01 | ChatGPT OAuth 認証成果物はコンテナ専用の Codex home にのみ保存する。 |
| SEC.03.02 | Codex の会話履歴、session index、state DB はコンテナ専用の Codex home にのみ保存する。 |
| SEC.03.03 | bot 運用停止やコンテナ再作成時に、Codex home をまとめて破棄または初期化できるようにする。 |

### 【通信境界】

| ID | 要求文 |
|---|---|
| NET.01 | Codex の通信先は公開 Web 取得に必要な外向き通信に限定する。 |
| 理由 [要約] | Docker を使って「ポートが見えないようにしたい」ため。 |
| NET.01.01 | Codex から `localhost`、private IP、LAN、Docker 管理系 endpoint、ホスト公開ポートへ到達させない。 |
| NET.01.02 | bot の通常運用でコンテナ受信用ポート公開を必須にしない。 |

### 【自己改造制御】

| ID | 要求文 |
|---|---|
| AUTH.04 | リポジトリ改変を伴う自己改造タスクは owner/admin の明示依頼かつ管理者限定 command により開始した制約緩和セッションでのみ受け付ける。 |
| 理由 [要約] | 通常会話と bot 運用を維持したまま、将来の自己改造余地だけを管理者専用に残すため。 |
| AUTH.04.01 | 通常会話、雑談、URL ingest、knowledge thread follow-up、通常の admin_control 会話からは自己改造を開始しない。 |
| AUTH.04.02 | 自己改造要求は owner/admin の明示依頼時だけ受理し、管理者限定 command による active override が無い限り read-only で扱う。 |
| AUTH.04.03 | 非管理者の要求では Codex の sandbox を workspace-write へ切り替えない。 |

### 【知見保持】

| ID | 要求文 |
|---|---|
| MEM.01 | bot は、取得した URL 内容と生成した要約・タグを、機密区分を保ったまま保持する。 |
| 理由 [要約] | 「LLMに与える情報は無駄じゃないものは多いほうが良い」「漏れないようにする必要がある」 |
| MEM.01.01 | 保持対象ごとに機密区分を記録する。 |
| MEM.01.02 | 応答時に参照対象を機密区分で絞り込む。 |
| MEM.01.03 | 同等以下の公開範囲でのみ保持データを再利用する。 |

### 【失敗時】

| ID | 要求文 |
|---|---|
| ERR.01 | URL 内容の取得または応答処理に失敗したとき、失敗理由と再試行予定を利用者に通知する。 |
| 理由 [引用] | 「もちろん、失敗理由と再試行予定を出す形」 |
| ERR.01.01 | 処理失敗の理由を生成する。 |
| ERR.01.02 | 再試行予定を生成する。 |
| ERR.01.03 | 生成した内容を対象の会話場所に通知する。 |

### 【ペルソナ】

| ID | 要求文 |
|---|---|
| PER.01 | bot は、参加者に親しみやすい秘書的存在として、一貫した話し方と振る舞いで応答する。 |
| 理由 [要約] | 「親しみやすいようにペルソナを設定した」「AIアシスタントというか秘書だな秘書的存在だと思った方がいい」 |
| PER.01.01 | 雑談用チャンネルで親しみやすく落ち着いた口調で応答する。 |
| PER.01.02 | 知見共有や質問応答でも、内容の正確さを保ったまま秘書的な話し方で応答する。 |
| PER.01.03 | 会話場所が変わっても、同一の bot と分かる一貫した受け答えを維持する。 |
| PER.01.04 | 馴れ馴れしすぎる表現、幼すぎる表現、過度なネットスラング、強い語尾キャラクター表現を用いない。 |
| PER.01.05 | 利用者ごとの継続的な関係性や個別の呼び方を前提とした応答は行わない。 |

- 「自分がPC起動しているときだけローカルで動かし、Codex App Server経由で動作する想定」

- 「GitのBot機能側でAdministrator権限を持っている人だけが使えるコマンドを用意して、そのコマンドはCodexをWorkspace-editを許可し、それ以外の場合はRead-onlyで運用すれば権限管理者だけが自己改造指示を出せる仕様がセキュアに作れる」
- 「全てのチャンネルのURLでいい」
- 「LLMに与える情報は無駄じゃないものは多いほうが良い」
- 「投稿場所と投稿内容の機密区分で良いと思う」

## 確認メモ

- Discord の権限仕様、チャンネル可視性、スレッド可視性、履歴取得範囲に照らして、`同等以下の公開範囲でのみ再利用する` 制御がどこまで実装しやすいかは確認が必要。
- bot 停止中の投稿を起動後に過去ログから取りこぼしなく再取得できる条件は、Discord API の取得方法と履歴保持前提の確認が必要。
